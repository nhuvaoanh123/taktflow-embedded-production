# SC E2E Format Gap

**Date:** 2026-03-21
**Severity:** MAJOR
**Found by:** SC integration test (Layer 6)

## Finding

SC_Status (0x013) uses a **custom E2E format** that does NOT match AUTOSAR E2E Profile P01 used by the BSW stack:

| Field | BSW E2E P01 | SC Custom |
|-------|------------|-----------|
| Byte 0 | [Counter:4 \| DataId:4] | AliveCounter (8-bit) |
| Byte 1 | CRC over payload[2:]+DataId | CRC over bytes 0,2,3 |
| Counter bits | 4-bit (0-15) | 8-bit (0-255) |
| DataId in CRC | Yes | No |
| CRC input | payload[2:N] + DataId | bytes 0, 2, 3 |

## Impact

1. CVC's Com E2E Check will reject SC_Status frames (CRC mismatch)
2. Python DBC encoder cannot produce valid SC_Status frames
3. DBC shows E2E signals for SC_Status but firmware doesn't follow that format
4. Any receiver using BSW E2E P01 will discard SC_Status

## Root Cause

SC was designed independently (TMS570, no AUTOSAR BSW) with its own minimal E2E before the shared BSW E2E module existed. The DBC was updated to show E2E signals for documentation, but the SC firmware was never aligned.

## Options

### A) Align SC to BSW E2E P01 (recommended)
- Change sc_monitoring.c to use [counter:4|dataId:4] format
- Change CRC input to payload[2:]+dataId
- Reduce alive counter to 4-bit
- ~2 hours, requires SC unit test updates

### B) Add SC-specific E2E decoder to CVC
- CVC's Swc_CanMonitor checks SC_Status with custom E2E
- Breaks the "single E2E format" principle
- More complex, harder to audit

### C) Remove E2E from SC_Status
- SC_Status becomes QM (no E2E protection)
- Simplest but weakens safety case
- SC relay kill is the safety-critical path, not SC_Status TX

## Recommendation

Option A. SC_Status should use the same E2E as everything else. The SC is simple enough that changing 4 lines in sc_monitoring.c is low risk.
